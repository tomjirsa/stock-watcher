How should the TA layer be structured?

Three approaches — all use pandas-ta as the library. The difference is where indicator math lives.

A

Inline — each layer computes its own

Strategies call pandas-ta directly. The Streamlit app also calls pandas-ta directly for chart rendering. Minimal structural change.

src/strategies/macd.py → df.ta.macd()
src/strategies/bbands.py → df.ta.bbands()
app/main.py → df.ta.macd(), df.ta.bbands(), df.ta.ema()
⚠ Same math in two places

Pros

  • Fewest new files
  • No new abstractions

Cons

  • Indicator math duplicated
  • Harder to unit-test in isolation
B

Shared src/indicators.py module ★ Recommended

A single module with pure functions: compute_macd(), compute_bbands(), compute_ema(), compute_rsi(). Strategies and the Streamlit app both import from it.

src/indicators.py ← math lives here
src/strategies/macd.py → from src.indicators import compute_macd
src/strategies/bbands.py → from src.indicators import compute_bbands
app/main.py → from src.indicators import compute_macd, compute_ema …
✓ Single source of truth

Pros

  • DRY — change once, both layers update
  • Easy to unit-test indicators in isolation
  • Natural place to add new indicators

Cons

  • One extra file
C

Pre-compute on StockData

StockData gets an indicators field. The Scanner (or DataService) computes all indicators before running strategies. Strategies just read stock.indicators['macd'].

src/models.py → StockData(indicators=…)
src/scanner.py → stock.indicators = compute_all(prices)
src/strategies/*.py → stock.indicators['macd_line']
app/main.py → fetches separately (UI has own data pipeline)
⚠ UI still needs its own compute path

Pros

  • Strategies don't import TA library directly

Cons

  • Computes all indicators even when unused
  • UI still needs a separate compute path
  • DataService/Scanner scope grows

Library: pandas-ta in all three. Pure Python, no C binaries, native pandas DataFrame integration (df.ta.macd() etc.).
